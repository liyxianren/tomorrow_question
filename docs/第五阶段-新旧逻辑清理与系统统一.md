# 第五阶段：新旧逻辑清理与系统统一

> 文档定位：第五阶段重构方案文档。基于前四阶段设计，清理当前代码中新旧逻辑并存的问题。
> 范围：议会大厅、研究院、海外扩张、生产系统的逻辑统一。
> 依据：[第一阶段-市场与生产机制](./第一阶段-市场与生产机制.md)、[第二阶段-政府决策与政治改革](./第二阶段-政府决策与政治改革.md)、[第三阶段-科技树与研究突破](./第三阶段-科技树与研究突破.md)、[第四阶段-军事与殖民地机制](./第四阶段-军事与殖民地机制.md)。

---

## 一、问题诊断

### 1.1 根本问题

当前代码是在 1.0 版本基础上**扩展**而非**重构**，导致新旧逻辑并存，游戏无法闭环。

### 1.2 四大模块问题清单

#### 模块 A：生产系统

| 问题 | 位置 | 影响 |
|------|------|------|
| `production.json` 同时定义新旧商品 | 后端配置 | 系统不知道该用哪套逻辑 |
| 前后端代码仍用 `goodsId` 区分商品 | 全栈 | 旧逻辑未清除 |
| 测试代码大量使用旧商品（coal, steel, grain） | 测试 | 测试无法验证新逻辑 |

**旧逻辑（1.0）：**
- 多种具体商品：棉花、矿产、茶叶、煤炭、粮食、钢铁、丝绸、石油、橡胶
- 每种商品有自己的价格、需求、生产路线
- 生产订单按商品分类（goodsId）

**新逻辑（第一阶段文档）：**
- 统一商品模型：只有"统一商品"（phase1_goods）
- 生产方式决定效率：手工业、机械化、蒸汽、电气化
- 原材料 → 商品，不分种类
- 价格由供需决定

#### 模块 B：议会大厅

| 问题 | 位置 | 影响 |
|------|------|------|
| 旧"科技点拨款"逻辑仍在 | GovernmentPanel.tsx | 财政→科技点的旧路径未移除 |
| 旧"执政议程"逻辑仍在 | GovernmentPanel.tsx | 策略选择是 1.0 扩展，不是第二阶段的改革+政策 |
| 新"改革/政策/思潮"逻辑未实现 | 全栈 | 第二阶段核心功能缺失 |

**旧逻辑（1.0 扩展）：**
- 科技点拨款（财政→科技点）
- 执政议程（策略选择）
- 国家权柄（一次性能力）

**新逻辑（第二阶段文档）：**
- 改革路径（自由/平等/民族三路径，23 项改革）
- 政策执行（公开募股、社会福利、资本税等）
- 思潮信号（平等主义/自由主义/民族主义，与经济挂钩）
- 行政能力（20$/行政力，改革消耗行政力）

#### 模块 C：研究院

| 问题 | 位置 | 影响 |
|------|------|------|
| 旧"预算直购科技"逻辑仍在 | decision.py / workspaces.py | 扣财政→即时解锁的旧路径未移除 |
| 新"研究设施+进展+突破"逻辑未实现 | 全栈 | 第三阶段核心功能缺失 |

**旧逻辑（1.0 扩展）：**
- 预算直购科技（扣财政→即时解锁）

**新逻辑（第三阶段文档）：**
- 研究设施建设（12 预算/设施）
- 研究进展累积（每回合 +1/设施）
- 科技突破掷骰子（D10，阈值判定）
- 三支科技链（电气/机械/蒸汽）

#### 模块 D：海外扩张

| 问题 | 位置 | 影响 |
|------|------|------|
| 旧"军事点消耗"逻辑仍在 | military_actions.json / decision.py | 简单军事点消耗的旧路径未移除 |
| 新"海军部署+封锁+陆军远征"逻辑未实现 | 全栈 | 第四阶段核心功能缺失 |

**旧逻辑（1.0 扩展）：**
- 军事点消耗
- 简单殖民地收入

**新逻辑（第四阶段文档）：**
- 海洋节点海军部署
- 海上封锁判定
- 陆军沿航线远征夺取殖民地
- 区域独立度变化 + 驻军镇压

## 二、重构方案

### 2.1 执行顺序

```
阶段五重构顺序：
1. 生产系统清理（基础层，影响其他模块）
2. 议会大厅重构（依赖生产系统的供需模型）
3. 研究院重构（依赖财政池）
4. 海外扩张重构（依赖海军/陆军系统）
5. 端到端验证（游戏闭环）
```

### 2.2 详细任务清单

- [ ] 清理 `decision.py`：移除 `goodsId` 相关逻辑，改为统一商品
- [ ] 清理 `settlement.py`：移除按商品分类的销售逻辑，改为统一商品销售
- [ ] 清理 `workspaces.py`：移除 `productionOptions` 中的商品分类，改为产能结构视图
- [ ] 清理 `factory_economy.py`：移除旧商品价格计算，改为供需定价

**前端改动：**
- [ ] 清理 `panelGlossary.ts`：移除旧商品标签（coal, cotton, grain 等）
- [ ] 清理 `FactoryPanel.tsx`：移除按商品分类的生产 UI，改为产能结构+原材料分配
- [ ] 清理 `MarketPanel.tsx`：移除按商品分类的销售 UI，改为统一商品投放
- [ ] 清理 `gameSnapshotFixtures.ts`：移除旧商品测试数据

**测试改动：**

- [ ] 更新 `test_game_state_models.py`：移除旧商品断言
- [ ] 更新 `test_m3_replace_rules.py`：移除旧商品逻辑

#### 任务 2：议会大厅重构

- [ ] 整体：15 回合完整跑通，无报错，无逻辑断裂

## 三、风险与注意事项

### 3.1 数据迁移风险

- 旧商品数据（goods_stock, productionOrders 等）需要迁移到新模型
- 建议：先备份现有数据，再执行迁移脚本

### 3.2 测试覆盖风险

- 旧测试用例大量依赖旧商品逻辑
- 建议：先更新测试用例，再执行重构

### 3.3 前端 UI 兼容风险

- 旧 UI 组件可能与新逻辑不兼容
- 建议：先重构 UI 组件，再对接新后端

### 3.4 配置文件一致性风险

- 多个配置文件（production.json, politics.json, reforms.json 等）需要同步更新
- 建议：建立配置文件版本管理，确保一致性
