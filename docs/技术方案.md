# 技术方案

## 1. 系统目标

本项目首版要交付一个由服务端主导状态真相、前端负责提交意图与展示结果的多人网页策略游戏。  
技术方案必须服务于下面这套正式规则：

`国家决策 -> 市场出售 -> 财政结算`

核心经济链路固定为：

`国内销售额 + 海外销售额 -> 国家收入 -> 按比例分配到三池 -> 驱动下一回合决策`

## 2. 架构边界

### 后端职责

- 管理房间、玩家会话、开局与恢复
- 持有唯一可信的 `GameSnapshot`
- 负责三阶段规则执行与财政结算
- 负责国家收入分配、预算池扣减、点数购买、比例变化
- 负责实时广播与对局结束判定

### 前端职责

- 展示房间、对局、结算与排行榜
- 根据当前阶段渲染对应工作台
- 组装玩家输入并提交给后端
- 展示国内销售额、海外销售额、国家收入、预算池变化、比例变化

### 持久化与部署职责

- SQLite 作为首版单实例持久化方案
- Flask 提供 HTTP 与 Socket 服务
- React/Vite 负责前端构建与页面渲染
- Docker + Zeabur 负责单容器交付

| 海外销售额 | `overseasSalesRevenue` |

## 4. 核心状态模型

### GameSnapshot

`GameSnapshot` 是服务端唯一真相，至少需要包含：

- 对局元信息
  - `gameId`
  - `round`
  - `phase`
  - `phaseDeadlineAt`
- 当前收入结构
  - `incomeAllocationRatio`
  - `budgetPools`
- 玩家经济与点数状态
  - `domesticSalesRevenue`
  - `overseasSalesRevenue`
  - `nationalIncome`
  - `techPoints`
  - `militaryPoints`
  - `cumulativeNationalIncome`
- 工业状态
  - 工厂列表
  - 产能
  - 库存
  - 工业路线等级
- 国家结构状态
  - 已激活的科技策略
  - 已激活的军事策略
  - 已激活的扶持类政策
  - 当前比例修正

### 阶段工作区

运行时只暴露三类工作区：

1. `DecisionPhaseWorkspace`
2. `MarketPhaseWorkspace`
3. `SettlementPhaseWorkspace`

每个工作区都只暴露当前阶段所需的信息，避免把结算与历史中间量泄漏成前端可编辑状态。

## 5. 三阶段执行模型

### 5.1 国家决策

后端接收三类玩家输入：

- `factoryPlan`
- `domesticMarketPlan`
- `governmentPlan`

执行逻辑：

- 校验三个预算池是否足额
- 校验科技点、军事点购买与消耗是否合法
- 记录工厂建设、生产、消费扶持、科技策略、军事策略、政策动作
- 更新当前回合可用于出售的库存、价格修正、海外市场能力与当前比例

### 5.2 市场出售

后端接收卖货方案，核心包括：

- 商品卖向
- 商品卖量
- 国内 / 海外分配

执行逻辑：

- 校验库存是否足够
- 校验市场承接能力
- 计算卖出数量与未卖出库存
- 写入 `domesticSalesRevenue`、`overseasSalesRevenue`
- 汇总 `nationalIncome`

### 5.3 财政结算

财政结算为系统自动阶段，不接受玩家输入。

执行逻辑：

- 汇总本回合国家收入
- 按 `incomeAllocationRatio` 分配到三个预算池
- 清理本回合临时工作区
- 累加 `cumulativeNationalIncome`
- 推进到下一回合或结束对局

## 6. 模块职责建议

### 后端模块

- `room`
  - 房间与准备逻辑
- `session`
  - 玩家恢复与身份绑定
- `game_state`
  - 快照、工作区、派生视图
- `rules`
  - 国家决策、市场出售、财政结算三阶段规则
- `settlement`
  - 回合推进、最终排名、阶段摘要
- `realtime`
  - Socket 广播、阶段计时、自动补交
- `persistence`
  - SQLite 存取与恢复

### 前端模块

- `pages`
  - Home / Lobby / Room / Game / Settlement
- `features/game`
  - 对局控制器、阶段表单、工作台视图模型
- `components/game`
  - 工厂、消费、政府、出售、结算等面板
- `services/api`
  - HTTP / Socket 客户端与恢复

## 7. 前后端交互原则

- 规则判定统一在后端执行
- 前端只提交“意图”，不做本地真结算
- 财政结算阶段由服务端自动推进
- 所有关键结果都必须可序列化进快照，保证刷新恢复

## 8. 风险与后续扩展

### 当前风险

- 三池独立预算会增加 UI 解释成本
- 当前比例与政策联动较多，若前端反馈不足，玩家容易混淆
- 市场出售与财政结算耦合较紧，契约必须先稳定

### 可扩展方向

当前模型天然支持后续加入：

- 人口
- 税收
- 劳动力
- 更细粒度海外市场
- 更复杂的科技树

扩展原则是不破坏“国家收入 -> 按比例分账 -> 三池驱动决策”的主链路。
