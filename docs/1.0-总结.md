## 一、1.0 产品定义

### 1.1 游戏定位

19 世纪工业化列强策略游戏。5 人固定国家、15 回合同步提交、统一结算的网页策略游戏。

玩家扮演英国 / 法国 / 普鲁士 / 奥地利 / 俄罗斯之一，通过工业建设、国内消费经营、科技与军事策略、海外市场拓展提升国家收入。

### 1.2 回合主循环

```
国家决策 → 市场出售 → 财政结算
```

## 二、1.0 核心经济规则

### 2.1 国家收入

**国家收入 = 本回合国内销售额 + 本回合海外销售额**

这是唯一结算口径。不比谁囤钱最多，比的是整局累计创造了多少国家收入。

### 2.2 三个独立预算池

每回合财政结算后，国家收入按当前比例自动分配到三个独立预算池：

| 预算池 | 默认占比 | 职责 |
|--------|---------|------|
| 国内消费市场 | 30% | 扶持国内消费、提升国内市场承接能力、提高收购价格 |
| 工厂 | 30% | 安排生产、支付用工成本、开办新厂、扩产、升级工业路线 |
| 政府财政 | 40% | 购买科技点/军事点、执行科技/军事策略、执行国家政策 |

初始分配比例固定为 **3 : 3 : 4**。三池独立，不可自由挪用。

### 2.3 三阶段职责

| 阶段 | 类型 | 内容 |
|------|------|------|
| 国家决策 | 玩家主动 | 工厂决策 + 国民消费决策 + 政府财政决策 |
| 市场出售 | 玩家主动 | 决定卖向（国内/海外）、卖量、压仓库风险 |
| 财政结算 | 系统自动 | 计算收入 → 汇总国家收入 → 按比例分配三池 → 推进回合 |

> 关键区分：国民消费决策消耗「国内消费市场」池，政府扶持内需政策消耗「政府财政」池——两者来源不同。

### 2.4 政府策略与国家结构变化

政府财政动作不仅产生即时效果，还会**永久改变国家收入分配比例**。

示例：执行海外扩张后，比例可从 `3:3:4` 变为 `2.8:3:4.2`，使后续回合更偏财政与海外方向。

反过来，国家补贴政策和扶持国内市场政策可将结构拉回内需方向。

### 2.5 科技点与军事点

- **科技点**：用于研发新产品、新工艺和更高价值工业路线
- **军事点**：用于开拓海外市场、扩张海外版图、提升海外控制力

点数需先用政府财政购买，再消耗执行动作。

### 2.6 胜负判定

**15 回合累计国家收入最高者获胜。**

## 三、1.0 五国设计

五个国家差异重点不在"按钮完全不同"，而在于**效率倾向**：

| 国家 | 倾向 | 开局配比 |
|------|------|---------|
| 英国 | 海外扩张与财政强化 | 国内 2.5 / 工厂 3.0 / 财政 4.5 |
| 法国 | 均衡经营 | 国内 3.5 / 工厂 3.0 / 财政 3.5 |
| 普鲁士 | 工业推进 | 国内 2.0 / 工厂 4.0 / 财政 4.0 |
| 奥地利 | 稳态军政 | 国内 3.0 / 工厂 3.0 / 财政 4.0 |
| 俄罗斯 | 中后期积累与爆发 | 国内 3.5 / 工厂 3.5 / 财政 3.0 |

各国初始意识形态等级、初始商品、初始外交区域、军事/研究设施均不同（详见 `backend/config/balance/countries.json`）。

### 数值规划（五国差异化方向）

- 英国：海外扩张效率更高
- 法国：三池转化较均衡
- 普鲁士：工厂扩张和工业推进效率更高
- 奥地利：政府策略更稳
- 俄罗斯：中后期累积能力更强

## 四、1.0 交付范围

### 4.1 必须交付

- 大厅、房间、准备、开局、对局、结算完整链路
- 15 回合完整闭环
- 三阶段同步提交与统一结算
- 关键数值展示、阶段摘要、排行榜、日志、刷新恢复
- Socket 实时同步、超时补交、Bot 自动提交
- 5 人固定房间可正常开局并跑满 15 回合

### 4.2 暂不纳入

- 账号体系
- 匹配、观战、历史战绩
- AI 补位之外的复杂 PVE
- 多实例扩容
- 人口、税收、劳动力等更深层宏观经济系统
- 美术与动画精修

## 五、1.0 技术架构

### 5.1 架构边界

- **后端**（Flask + SQLite）：管理房间/玩家会话、持有唯一可信 GameSnapshot、执行三阶段规则与财政结算、Socket 广播
- **前端**（React/Vite）：展示房间/对局/结算/排行榜、渲染阶段工作台、组装玩家输入提交、展示国家收入和预算池变化
- **部署**：Docker + Zeabur 单容器

### 5.2 核心数据模型

**GameSnapshot**（服务端唯一真相）包含：
- 对局元信息（gameId, round, phase, phaseDeadlineAt）
- 每个玩家的 NationalState（domesticSalesRevenue, overseasSalesRevenue, nationalIncome, cumulativeNationalIncome, incomeAllocationRatio, budgetPools, techPoints, militaryPoints, factories, inventory, activePolicies, overseasMarketAccess）
- 阶段工作区（DecisionPhaseWorkspace / MarketPhaseWorkspace / SettlementPhaseWorkspace）
- 排行榜

**旧语义（已清退）**：treasuryBalance, cumulativeTreasuryIncome, pendingSettlementIncome, domesticStrength, industryStrength, statecraftStrength, fundPools

### 5.3 术语对照

| 中文 | 技术字段 |
|------|---------|
| 国家收入 | nationalIncome |
| 国内销售额 | domesticSalesRevenue |
| 海外销售额 | overseasSalesRevenue |
| 累计国家收入 | cumulativeNationalIncome |
| 收入分配比例 | incomeAllocationRatio |
| 国内消费市场 | budgetPools.domesticMarket |
| 工厂 | budgetPools.factory |
| 政府财政 | budgetPools.governmentFiscal |
| 科技点 | techPoints |
| 军事点 | militaryPoints |

### 5.4 接口设计

| 接口 | 用途 |
|------|------|
| POST /api/v1/rooms | 创建房间 |
| POST /api/v1/rooms/{id}/join | 加入房间 |
| POST /api/v1/rooms/{id}/ready | Ready/Cancel |
| POST /api/v1/rooms/{id}/recover | 恢复会话 |
| GET /api/v1/games/{id} | 获取 GameSnapshot |
| POST /api/v1/games/{id}/phases/decision/submit | 提交国家决策 |
| POST /api/v1/games/{id}/phases/market/submit | 提交市场出售 |
| GET /api/v1/games/{id}/settlement | 获取最终结算 |

Socket 事件：game.snapshot / game.phase_started / game.phase_settled / game.finished / room.updated

### 5.5 状态流转

1. **回合开始** → decision 阶段，加载三池、点数、工厂库存
2. **国家决策结算** → 校验三池余额、扣减花费、更新工厂/消费/策略/比例
3. **市场出售结算** → 分配库存、计算国内/海外销售额、汇总国家收入
4. **财政结算** → 按比例分账到三池、累加 cumulativeNationalIncome、推进回合或结束

## 六、1.0 前端设计

### 6.1 页面结构

Home → Lobby → Room → Game → Settlement

### 6.2 Game 页设计

**顶部信息带**：当前回合/总回合、当前阶段、当前国家、收入分配比例、国家收入摘要

**核心资源带**：三池（国内消费市场 / 工厂 / 政府财政）+ 科技点 + 军事点（并列展示，不做主钱包+附属标签）

**阶段工作台**：根据阶段切换三类主面板——国家决策台 / 市场出售台 / 财政结算台

**国家决策台**固定拆为三区：
- 工厂决策（工厂预算，产线排程，建设改造，工业研究）
- 国民消费决策（国内预算，民生政策，国内市场容量）
- 政府财政决策（政府财政，科技点拨款/购买，政策与国家能力）

**五大入口**（主地图）：工业区 / 市民广场 / 议会厅 / 军事要塞 / 研究院

### 6.3 文案规范

前台阶段固定为：国家决策 / 市场出售 / 财政结算。禁止出现旧四阶段名称。

必须区分：
- 国民消费决策（花国内消费市场）≠ 政府扶持内需政策（花政府财政）
- 国内销售额（卖到国内赚到的钱）≠ 国内消费市场（收入分配后的预算池）

## 九、1.0 后续演进文档

### 9.1 2026-04-27 经济机制重构会议

**会议结论**：当前 Demo 的经济底层模型与客户预期存在根本差异。

| 维度 | 当前 Demo | 会后目标 |
|------|----------|---------|
| 商品体系 | 多种具体商品（钢铁、茶叶等） | 抽象为原材料与商品，通过生产效率体现差异 |
| 价格形成 | 随机/事件驱动 | 供给-需求-消费池资金形成动态价格 |
| 工厂作用 | 偏生产/扩产 | 既决定供给也创造需求 |
| 验收状态 | 可作 UI/流程 Demo | 不能作为核心经济机制验收版本 |

会后优先重构：工业区（原材料→商品抽象模型）、市民广场（供需关系与价格形成）。

### 9.2 目标经济模型（供需价格机制）

核心公式：
- 商品供给 = 原材料数量 × 生产效率系数
- 均衡价格 = 上回合消费池资金 / 总需求
- 价格波动：供给<需求→价格上涨，供给>需求→价格下跌

### 9.3 V2 优化文档

阶段 1-4 完成后发现的可玩性问题及五大特性计划：

| 编号 | 特性 | 状态 |
|------|------|------|
| F1 | 国际事件系统（30 事件） | ✅ 已完成 |
| F2 | 动态市场定价 | 待开发 |
| F3 | 意识形态与改革 | ✅ 已完成 |
| F4 | 国家专属能力卡 | 待开发 |
| F5 | 渐进解锁科技树（瘟疫公司风格） | 待开发 |

F5 核心设计：商品和策略不再全部可用，需花科技点逐步解锁。线性链式解锁，15 回合内不可能全部解锁，必须选择发展路线。包含三大科技链（工业/政策/消费）+ 详细实现步骤。

### 9.4 V3 优化文档（军事模块）

将军事升级为独立游戏模块：
- 地图新增军事要塞建筑
- 专属军事决策面板（造舰、征兵、建交、远征）
- 海外区域准入状态可视化
- 从议会厅移除军事策略

实现涉及：新建 military_actions.json、PlayerState 新增 established_diplomacy、区域准入逻辑改造、前端军事工作区类型与 UI。
