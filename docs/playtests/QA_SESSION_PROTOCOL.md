# Tomorrow Question 玩家视角测试闭环文档

> 每次模型记忆重置后，只要任务涉及真实玩家测试、1-15 回合流程、按钮覆盖、数值验证或 bug 修复，先阅读本文件，再开始操作。

## 固定入口

- 测试方式：必须使用 Computer Use 控制真实浏览器页面，不能用后端 API 代替玩家点击。
- 默认本地服务：后端 `http://127.0.0.1:5001/healthz`，前端 `http://127.0.0.1:5173`。
- 每次完整测试都新建 markdown 记录，路径放在 `docs/playtests/`。
- 每次发现问题先记录，不直接修，除非 P0 阻断继续测试。
- 每次修复后必须回到真实页面复验，而不是只跑单元测试。

## 下一次完整测试目标

一次性从第 1 回合跑到第 15 回合终局，覆盖所有主要按钮逻辑，并记录每个按钮对数值、状态、提交、下回合生效的影响。

本轮不是“快速通关”，而是“玩家视角覆盖矩阵”：

- 每个主建筑都要打开并操作。
- 每类可选按钮、撤回按钮、跳过按钮、已检查按钮都要覆盖。
- 每个关键数值变化都要记录点击前、点击后、提交后、下一回合。
- 每个锁定/不可用按钮都要记录锁定理由是否准确。
- 每个系统至少验证一次刷新/重开页面后的恢复表现。

## 覆盖矩阵

| 模块 | 必测按钮/状态 | 必测数值 | 必测持久化 |
| --- | --- | --- | --- |
| 工厂 | 生产加减、清零、最大、工厂增加、产业升级、材料购买、撤回、锁定说明 | 工厂预算、原材料、总工厂池、闲置工厂、库存、国内需求 | 扩建和升级是否本回合生效；材料购买是否本回合可投料；单个行业不应再有上限 |
| 政府 | 贸易促进、改革路径、常规政策、国家能力、行政力、跳过、已检查、撤回 | 政府财政、行政力、政策占用、思潮、海外容量 | 行政力是否每回合刷新；政府财政是否单池；不再出现政策专项额度 |
| 市场 | 国内投放加减、海外投放加减、清零、最大、提交、市场竞争 | 库存、国内需求、定价软上限、国内价格、海外容量、区域固定价、预计收入 | 贸易促进、事件、竞争容量、地区封锁是否影响出售阶段 |
| 军事 | 征募陆军、建造舰队、地区舰队部署、撤回、锁定说明 | 政府财政、陆军、舰队、地区封锁、海外区域状态 | 建造舰队是否下回合真实 +1；地区封锁是否跨回合保留；不再出现海军演练或建交入口 |
| 研究 | 建立研究院、撤回研究院、选择研究目标、锁定科技、跳过、已检查 | 政府财政、研究设施、当前研究进度、阈值、突破状态 | 研究设施是否下回合增加进度；研究目标是否延续；首研/后发规则是否正确 |
| 提交流程 | 当前不可提交、阻断原因、确认提交、阶段跳转、倒计时 | 步骤状态、资源条、结算收入、累计收入 | 刷新后草稿是否恢复；提交后是否进入正确阶段；第 15 回合是否进终局 |
| 终局 | 返回大厅、回到房间、排名、日志、关键转折 | 最终排名、累计收入、回合数、终局日志分类 | 终局页不漏 i18n key；不出现 raw 英文系统日志 |

## Bug 等级

- `P0`：阻断继续测试、提交失败、页面崩溃、数据损坏、云端/本地无法进入关键阶段。
- `P1`：核心规则错误、数值前后端不一致、按钮看似成功但下回合不生效。
- `P2`：重要体验/文案/反馈错误，会误导玩家但不阻断流程。
- `P3`：可读性、轻微文案、非核心显示问题。

## 问题闭环规则

所有问题都必须进入同一个文档的“问题台账”，不要散落在对话里。

每个问题必须有：

- 唯一编号。
- 回合和阶段。
- 复现路径。
- 期望结果。
- 实际结果。
- 严重程度。
- 修复状态：`未修复` / `修复中` / `已修复待复验` / `已复验关闭`。
- 复验证据：测试命令或 Computer Use 页面证据。
