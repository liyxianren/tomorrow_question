# 接口与事件契约设计

## 1. 设计原则

接口契约要服务于三阶段正式规则：

`国家决策 -> 市场出售 -> 财政结算`

对外边界保持简单：

- HTTP 负责房间、恢复、快照与阶段提交
- Socket 负责阶段开始、阶段结算、快照更新、对局结束广播
- 规则真相始终由服务端掌握

## 2. 核心 HTTP 接口

### 房间与会话

- `POST /api/v1/rooms`
  - 创建房间
- `POST /api/v1/rooms/{roomId}/join`
  - 加入房间
- `POST /api/v1/rooms/{roomId}/ready`
  - Ready / Cancel Ready
- `POST /api/v1/rooms/{roomId}/recover`
  - 恢复玩家会话

### 对局快照

- `GET /api/v1/games/{gameId}`
  - 获取完整 `GameSnapshot`
- `GET /api/v1/games/{gameId}/settlement`
  - 获取最终结算摘要

### 阶段提交

- `POST /api/v1/games/{gameId}/phases/decision/submit`
  - 提交国家决策
- `POST /api/v1/games/{gameId}/phases/market/submit`
  - 提交市场出售

`settlement` 阶段为系统阶段，不提供玩家提交入口。

## 3. 阶段提交契约

### 3.1 国家决策提交

```json
{
  "factoryPlan": {
    "productionOrders": [],
    "expansionOrders": [],
    "upgradeOrders": [],
    "newFactoryOrders": []
  },
  "domesticMarketPlan": {
    "domesticMarketActions": []
  },
  "governmentPlan": {
    "pointPurchases": [],
    "strategySelections": []
  }
}
```

服务端校验：

- `budgetPools.factory` 是否足够覆盖工厂动作
- `budgetPools.domesticMarket` 是否足够覆盖国民消费动作
- `budgetPools.governmentFiscal` 是否足够覆盖点数购买和政府策略
- 点数消耗是否合法

### 3.2 市场出售提交

```json
{
  "saleOrders": [
    {
      "goodsId": "steel",
      "market": "domestic",
      "quantity": 3
    }
  ]
}
```

服务端校验：

- 库存是否足够
- 国内 / 海外市场承接能力是否足够
- 卖向是否合法

## 4. Socket 事件

### 服务端 -> 客户端

- `game.snapshot`
  - 广播最新 `GameSnapshot`
- `game.phase_started`
  - 新阶段开始
- `game.phase_settled`
  - 当前阶段结算完成
- `game.finished`
  - 对局结束，附最终排名摘要
- `room.updated`
  - 房间状态变化

### 事件载荷要求

#### `game.phase_started`

```json
{
  "gameId": "game_001",
  "round": 3,
  "phase": "market"
}
```

#### `game.phase_settled`

```json
{
  "gameId": "game_001",
  "round": 3,
  "phase": "settlement",
  "summary": {
    "nationalIncome": 18
  }
}
```

#### `game.finished`

```json
{
  "gameId": "game_001",
  "ranking": [
    {
      "playerId": "p1",
      "cumulativeNationalIncome": 146
    }
  ]
}
```

## 5. GameSnapshot 对外契约

对外返回的 `GameSnapshot` 至少包含：

```json
{
  "gameId": "game_001",
  "round": 1,
  "maxRounds": 15,
  "phase": "decision",
  "nationalStateByPlayer": {},
  "phaseWorkspace": {},
  "ranking": []
}
```

其中每个玩家的 `NationalState` 至少要暴露：

- `domesticSalesRevenue`
- `overseasSalesRevenue`
- `nationalIncome`
- `cumulativeNationalIncome`
- `incomeAllocationRatio`
- `budgetPools.domesticMarket`
- `budgetPools.factory`
- `budgetPools.governmentFiscal`
- `techPoints`
- `militaryPoints`
- 工厂、库存、策略摘要

## 6. 错误语义

阶段提交接口需要明确返回下列错误：

- `400`
  - 提交结构非法
- `401`
  - 玩家会话无效
- `403`
  - 非当前房间玩家或无权操作
- `409`
  - 阶段不匹配、重复提交、余额不足、点数不足、库存不足
- `410`
  - 对局已结束

## 7. 自动推进与超时补交

### 国家决策

- 允许等待所有玩家提交或触发超时自动补交
- 自动补交需要遵循当前预算池与点数约束

### 市场出售

- 同样允许等待所有玩家提交或触发超时自动补交
- 自动补交至少要保证默认卖货方案合法

### 财政结算

- 服务端自动执行
- 执行完成后广播新的快照与阶段变更

## 8. 恢复链路要求

玩家刷新页面后，恢复接口必须保证能重新拿到：

- 当前回合
- 当前阶段
- 三个预算池
- 当前比例
- 点数余额
- 工厂与库存状态
- 当前阶段工作区

恢复后的前端不应依赖本地缓存推断状态，必须以服务端快照为准。
