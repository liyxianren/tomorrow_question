## 一、用户需求骨架

### 1.1 四级市场独立度（§4.1）

| 等级 | 说明 | 代码 |
|------|------|------|
| 封闭 | 所有玩家均不能向其销售商品 | `RegionAccessLevel.CLOSED` |

```
财政池购买军力 → 部署海军到海洋节点 → 控制航线
                                    ↓
                              封锁竞争对手 → 垄断海外市场
                                    ↓
财政池购买陆军 → 沿航线远征 → 夺取殖民地

```

## 三、设计规范

### 3.1 海洋节点运行时状态

#### 3.1.1 模型

```python
@dataclass(slots=True)
class OceanNodeState:
    node_id: str
    naval_power: dict[str, int]  # player_id → 部署的舰队数
    blockade_controller: str | None  # 本回合封锁者 player_id

    def total_power(self) -> int:
        return sum(self.naval_power.values())
```

**初始化**：`factory.py` 从 `regions.json → oceanNodes` 创建，所有 `naval_power = {}`, `blockade_controller = None`。

**序列化**：加入 `GameSnapshot.to_payload() / from_payload()`。

#### 3.1.2 海洋节点拓扑（regions.json）

```
north_atlantic ←→ mediterranean ←→ indian_ocean ←→ pacific
     ↕                                    ↕              ↕
  americas                             africa       asia_pacific
                                     middle_east
```

每个 region 的 `requiredNodes` 定义了到达该区域必须经过的海洋节点：
- americas → [north_atlantic]
- africa → [mediterranean]
- middle_east → [mediterranean]
- asia_pacific → [indian_ocean]

### 3.2 海军部署（§4.2）

#### 决策阶段

在 `militaryPlan` 中新增字段：

```json
{
  "navalDeployment": {
    "north_atlantic": 2,
    "mediterranean": 1
  }
}
```

**校验**：
- 每个节点的部署数量 ≤ 玩家拥有的 `navy["fleets"]`
- 所有节点部署总和 ≤ 总舰队数
- 节点必须存在于 `oceanNodes` 配置中

**执行**（在 `_apply_military_plan` 中）：
- 写入 `snapshot.ocean_node_states[node_id].naval_power[player_id] = deployed_count`
- 从 `player_state.navy["fleets"]` 不扣除（舰队不被"消耗"，只是位置分配）

> **设计理由**：舰队是永久资产，部署只是位置分配。这避免了"部署一次就用掉"的挫败感。每回合重新分配位置即可。

### 3.3 海上封锁（§4.3）

#### 结算阶段（在 `settlement.py` 中，market 阶段之前）

```python
def _resolve_naval_blockade(snapshot, balance):
    threshold = balance.military.ocean_control_threshold  # 2
    for node in snapshot.ocean_node_states:
        if not node.naval_power:
            node.blockade_controller = None
            continue
        # 找出海军最多的玩家
        max_player = max(node.naval_power, key=node.naval_power.get)
        max_power = node.naval_power[max_player]
        # 需要 ≥ 阈值 且 > 所有其他玩家之和（或至少严格大于）
        others_power = node.total_power() - max_power
        if max_power >= threshold and max_power > others_power:
            node.blockade_controller = max_player
        else:
            node.blockade_controller = None
```

**封锁效果**（在市场销售和征服前检查）：
- 玩家要通过某海洋节点到达目标区域
- 如果该节点 `blockade_controller` 存在且 ≠ 当前玩家 → **不可通过**
- 封锁者本人不受影响

### 3.4 路线畅通检查（§4.2 后半）

```python
def _check_route_accessible(player_id, region_id, snapshot, balance) -> bool:
    """检查玩家是否可以通过航线到达目标区域。"""
    region_blueprint = balance.regions.region_blueprints.get(region_id)
    if region_blueprint is None:
        return False
    required_nodes = region_blueprint.required_nodes  # list[str]
    if not required_nodes:
        return True  # europe → 无需经过海洋节点
    ocean_nodes_by_id = {n.node_id: n for n in snapshot.ocean_node_states}
    for node_id in required_nodes:
        node = ocean_nodes_by_id.get(node_id)
        if node is None:
            continue
        if node.blockade_controller is not None and node.blockade_controller != player_id:
            return False  # 被其他人封锁
    return True
```

**使用场景**：
1. 市场阶段：海外销售前检查（已有一个 `is_region_accessible`，在此基础上叠加路线检查）
2. 征服阶段：攻击前检查路线

### 3.5 陆军夺取殖民地（§4.4）

#### 决策阶段

在 `militaryPlan` 中新增字段：

```json
{
  "conquestActions": [
    {"regionId": "africa", "army": 5}
  ]
}
```

**现有殖民逻辑保留**：`unlockColonization` + `colonizationActions` 是"和平殖民"路径（花 military_points 直接拿下无人区）。新增 `conquestActions` 是"武力征服"路径。

**两种路径的区别**：
| 路径 | 消耗 | 适用 | 路线要求 |
|------|------|------|---------|
| 和平殖民 | military_points（3点） | 无人区 | 需外交 + 解锁殖民 |
| 武力征服 | army（>0） | 无主区或他人殖民地 | 需路线畅通 |

**征服执行**：

```python
def _apply_conquest_plan(player_state, conquest_actions, snapshot, balance):
    for action in conquest_actions:
        region_id = action["regionId"]
        attacking_army = action["army"]
        
        # 1. 路线检查
        if not _check_route_accessible(player_state.player_id, region_id, snapshot, balance):
            continue
        
        # 2. 兵力检查
        total_army = sum(player_state.army.values())
        if attacking_army > total_army:
            continue
        
        # 3. 找到目标区域
        target_region = find_region(snapshot, region_id)
        region_blueprint = balance.regions.region_blueprints.get(region_id)
        if target_region is None or region_blueprint is None:
            continue
        
        # 4. 已有控制者？
        if target_region.controller is not None:
            # 夺取他人殖民地：需要 2x 守军
            defender_garrison = sum(target_region.garrison.values())
            if attacking_army < defender_garrison * 2:
                continue
            # 夺取成功：如果守军归零，更激进的可以给原主补偿
        else:
            # 无主区：需要最低军力
            min_army = region_blueprint.min_garrison or 1
            if attacking_army < min_army:
                continue
        
        # 5. 消耗陆军
        for unit_type in player_state.army:
            deducted = min(player_state.army[unit_type], attacking_army)
            player_state.army[unit_type] -= deducted
            attacking_army -= deducted
            if attacking_army <= 0:
                break
        
        # 6. 夺取成功
        target_region.controller = player_state.country.value
        target_region.garrison = {"infantry": action["army"]}  # 攻击部队成为驻军
```

> **关于"2 倍守军"的设计**：用户原意是"需要投入其他玩家在该区域驻扎陆军的 2 倍"，即攻击方需要 2× 守军兵力才能夺取。这是攻防不对称设计——防御方有地利优势。

### 3.6 区域独立度变化（§4.5）

#### 结算阶段（在 `settlement.py` 中，收入分配之后）

```python
def _apply_independence_progression(snapshot, balance):
    threshold = balance.military.independence_threshold  # 10
    
    for region in snapshot.region_states:
        if region.controller is None:
            region.independence = 0
            continue
        
        delta = 0
        
        # 1. 市场供需影响
        supply = sum(region.market_supply.values())
        demand = sum(region.resource_limit.values())
        if demand > 0:
            ratio = supply / demand
            if ratio > 1.5:      # 供给严重过剩
                delta += 2
            elif ratio > 1.0:    # 供给轻微过剩
                delta += 1
            elif ratio < 0.5:    # 供给严重不足
                delta += 2
            elif ratio < 1.0:    # 供给轻微不足
                delta += 1
        
        # 2. 驻军镇压
        garrison_total = sum(region.garrison.values())
        delta -= garrison_total
        
        # 3. 应用变化
        region.independence = max(0, region.independence + delta)
        
        # 4. 叛乱判定
        if region.independence >= threshold:
            region.controller = None
            region.garrison = {}
            region.independence = 0
```

**设计要点**：
- 供需失衡（无论过剩还是不足）都会推高独立度——殖民地的经济不稳定
- 驻军是唯一的镇压手段——倒逼玩家在殖民地保持军事存在
- 叛乱后果：失去殖民地，驻军清零

| 维度 | 和平殖民 | 武力征服 |

| 消耗 | 3 militaryPoints | army 兵力 |

## 七、风险与注意事项

1. **海军"部署"语义**：舰队不消耗，只分配位置。每回合自动根据上次分配保持？还是每回合重新部署？建议：每回合重新部署（决策阶段重新选择），简化状态管理。

2. **独立度与殖民地收入的取舍**：如果殖民地独立度太高叛乱，玩家失去控制器也就失去收入。这是设计意图——"掠夺一时爽，叛乱火葬场"。

3. **海洋节点状态持久化**：如果游戏支持存档/恢复，ocean_node_states 必须加入快照序列化。已包含在 Task 1。

4. **和平殖民路径是否废弃**：设计上两条路径并存。和平殖民花费 military_points（3 点），武力征服花费 army。和平路径更快（不需造船），但只能拿无主区。武力路径更灵活但需要海军投入。

5. **现有的 `military_points` 角色变化**：从"唯一军力货币"降级为"征兵副产品"。不影响现有测试（已有 15+ 个军事相关测试），只需扩展而非重写。
