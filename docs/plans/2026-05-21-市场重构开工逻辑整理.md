## 自洽性判断

国内市场的核心逻辑是自洽的：

- 需求来自产能结构，而不是玩家手动花钱买需求。
- 消费池资金参与价格计算，但当前没有“消费市场支出”动作。
- 供给等于需求时，国内总收入约等于消费池资金。
- 供给不足时单价上涨，但卖得少。
- 供给过剩时单价下降，但仍允许成交。
- 超过软上限后的低价出售用动态价格下限保护，避免负价格。
- 下回合价格不需要额外记忆字段；收入分配进入消费池后自然影响下回合均衡价。

海外市场也自洽：

- 海外区域默认开放，不再用建交决定准入。
- 海外价格只看区域固定价。
- 海外容量有限，政策、军事、市场竞争只改容量或可售上限。
- 封锁地区容量视为 0，并需要解释原因。

## 国内市场开工公式

### 变量

```text
capacityByMode = 当前国家各生产方式产能数量
demandCoefficient = 生产方式需求系数
D_base = Σ(capacityByMode[mode] × demandCoefficient[mode])
capacityBonus = 国内承接能力修正，来自政策/事件/能力
K = max(1, D_base + capacityBonus)
C = 本回合进入市场阶段时的消费池资金
Q = 玩家投入国内市场的商品数量
P0 = C / K
```

默认需求系数沿用用户原始需求：

| 生产方式 | 需求系数 |
| --- | ---: |
| 闲置工厂 | 1 |
| 手工业 | 2 |
| 机械化 | 3 |
| 蒸汽 | 4 |
| 电气化 | 5 |

### 单价

```text
当 Q = 0：
国内成交量 = 0
国内收入 = 0
预览可显示参考价 P0

当 0 < Q < K：
短缺率 = (K - Q) / K
国内单价 = P0 × (1 + 短缺率)

当 Q = K：
国内单价 = P0

当 Q > K：
过剩率 = (Q - K) / K
国内单价 = max(0.1 × P0, P0 × (1 - 过剩率))
```

统一实现：

```text
P0 = 消费池 / K
国内单价 = clamp(P0 × (2 - Q / K), 0.1 × P0, 2 × P0)
```

工程默认值：

| 参数 | 默认值 | 原因 |
| --- | ---: | --- |
| `minimumDomesticPrice` | `0.1 × P0` | 客户补充的成交价格下限。 |
| `domesticPriceCeiling` | `2 × P0` | 客户补充的成交价格上限。 |
| `domesticPriceBonus` | 保留 | 国内价格政策可继续影响国内价格，但不影响海外价格。 |
| `capacityBonus` | 保留 | 国内容量/承接能力政策改为影响软上限 `K`，不再硬拦截投放。 |

国内价格政策后的实现口径：

```text
国内单价 = clamp(国内供需单价 + domesticPriceBonus, 0.1 × P0, 2 × P0)
```

### 成交量和收入

```text
国内成交量 = Q
国内收入 = round(国内成交量 × 国内单价)
```

约束：

- `Q` 不能超过当前剩余库存。
- `Q` 不再被 `D_base` 或 `K` 硬拦截。
- 如果玩家超额投放，系统成交全部投放量，但用低价体现过剩。
- UI 必须显示 `Q / K`、短缺/均衡/过剩状态、单价、预计收入。

## 海外市场开工公式

区域固定价：

| 区域 | 固定价 |
| --- | ---: |
| 欧洲 | 8 |
| 美洲 | 7 |
| 亚太 | 6 |
| 中东 | 5 |
| 非洲 | 4 |

```text
区域可用共享容量 = 玩家海外容量剩余
竞争额外容量 = 玩家本回合在该区域竞争成功获得的额外容量
区域可售上限 = 区域可用共享容量 + 竞争额外容量
海外成交量 = min(投放量, 剩余库存, 区域可售上限)
海外收入 = 海外成交量 × 区域固定价
```

封锁：

```text
如果区域被对手封锁：
区域可售上限 = 0
海外成交量 = 0
海外收入 = 0
```

准入：

- 不检查建交。
- 不检查旧 `established_diplomacy`。
- 区域默认开放。
- 只检查封锁、库存、容量和数据合法性。

## 国内和海外库存关系

本轮采用统一商品库存：

```text
总投放量 = 国内投放量 + Σ海外投放量
总投放量 <= 当前商品库存
```

开发口径：

- 提交校验应拦截总投放量超过库存。
- 国内不硬拦截软上限。
- 海外允许投放量超过区域容量，但结算只成交容量内数量；前端应尽量用最大值按钮避免玩家误投。
- 国内和海外都从同一个 `phase1_goods` 库存扣减。

## 收入分账

用户原始需求初始比例是：

```text
消费 : 投资 : 财政 = 3 : 3 : 4
```

本轮开发应恢复为：

```text
总销售收入 = 国内收入 + 海外收入
消费池增加 = floor(总销售收入 × 30%)
工厂预算增加 = floor(总销售收入 × 30%)
政府财政增加 = 总销售收入 - 消费池增加 - 工厂预算增加
```

兼容政策：

- 如果政策改变分配比例，继续走现有 `incomeAllocationRatio`。
- 默认国家初始比例应调整为 `3:3:4`。
- 不再冻结消费池，也不再把消费份额重新分配给工厂和政府。

## 可直接开工的开发任务

1. 后端纯函数：
   - 新增 `calculate_effective_domestic_capacity()`
   - 新增或重写 `calculate_equilibrium_price(consumption_pool, effective_capacity)`
   - 重写 `calculate_domestic_price(equilibrium_price, allocation, effective_capacity)`，默认使用动态下限 `0.1×P0` 和上限 `2×P0`。
   - 让 `resolve_domestic_market()` 支持软上限全部成交。
2. 后端结算：
   - `market.py` 国内成交量不再 `min(demand, capacity)`。
   - `phase_submission.py` 删除需求/容量硬拦截，增加总库存校验。
   - `settlement.py` 恢复 3:3:4 分账。
   - 删除或停用 `market_price_adjustments` 对 Phase1 国内价格的直接影响。
3. Workspace：
   - 输出 `domesticSoftCap`、`consumptionPool`、`equilibriumPrice`、`minimumDomesticPrice`、`domesticPricePreview`、`surplusRate`、`shortageRate`。
4. 前端：
   - 国内投放上限 = 剩余库存。
   - 软上限显示为价格参考线，不是按钮禁用线。
   - 超额时显示“可成交，但单价下降”。
   - 预计收入和后端公式保持一致。
5. 测试：
   - `Q < K` 单价上涨。
   - `Q = K` 单价等于均衡价。
   - `Q > K` 单价下降但成交。
   - 极端过剩价格不低于 `0.1×P0`，极端短缺价格不高于 `2×P0`。
   - 国内投放超过软上限不报错。
   - 收入按 3:3:4 进入三池。
   - 海外固定价和容量不受国内公式影响。

## 不需要再问用户的问题

- 国内降价曲线和上下限：客户已补充为线性百分比，价格下限 `0.1×P0`，价格上限 `2×P0`。
- 闲置工厂是否参与需求：确认书和初始文档都支持，系数用 1。
- 海外市场价格：本轮已确认固定价，具体数值由我们设定。
- 市场竞争：本轮已确认只加容量，不加价格。
- 建交准入：本轮已确认删除。
- 收入分账：当前确认口径为 3:3:4。

## 仍需后续确认但不阻塞开工

- 国内价格政策是否用 `+N` 加价还是倍率。当前可先沿用 `+N`，后续 setting 调整。
- 海外容量默认值是否足够。当前可沿用 `baseOverseasCapacity = 5` 并在 setting 暴露。
- 如果客户后续要求“超额销售影响下回合市场状态”，再新增状态字段；当前不做。

## 开工结论

可以推进开发。第一批应先改后端纯函数和提交校验，再改 settlement 分账，最后同步前端预览和测试。最大风险不是需求不明，而是现有代码里仍有旧硬上限、固定基准价、消费池冻结和前端 clamp，这些需要一次性改掉，否则玩家看到的预览会和结算不一致。
