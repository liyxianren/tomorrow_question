# 2.0 阶段 3 实施计划

> 文档定位：阶段 3 可执行实施计划  
> 覆盖范围：恢复链、实时链、Bot 与超时补交链、测试与验收链  
> 前置条件：阶段 1 契约冻结 + 阶段 2 前端主链路完成  
> 不覆盖范围：新玩法开发、国家差异深化、前端主界面重构

## 1. 计划目标

本计划只服务一个目的：

**把已经完成 2.0 迁移的主页面和后端内核，正式接入恢复、实时、Bot、超时补交和测试链，形成可回归、可恢复、可自动推进的完整运行闭环。**

本计划是 [开发任务清单](../../开发任务清单.md) 中阶段 3 的唯一实施依据，也是 2.0 主闭环从“能跑页面”进入“能稳定运行”的收口文档。

## 2. 阶段 3 完成定义

阶段 3 完成后，系统必须满足以下条件：

1. 刷新恢复返回的 `activeSnapshot / activeGame / room / turnInputs / gameLogs` 全部按 2.0 契约工作
2. `gamePhaseStarted / gamePhaseSettled / gameFinished / gameSnapshotSync / gamePhaseTimer` 全部按 2.0 字段广播与消费
3. Bot 与超时补交只生成 `DecisionSubmission` 与 `MarketSubmission`
4. `settlement` 保持系统自动推进，无玩家提交、无 Bot 提交
5. 后端测试、前端运行时测试、本地验收和浏览器主流程按 2.0 完整通过

## 3. 阶段范围与边界

### 3.1 本阶段覆盖

- 会话恢复与快照恢复
- Socket 广播与 phase timer
- Bot 自动提交
- timeout 自动补交
- 前端 runtime 恢复与事件应用
- 后端、前端、本地验收、浏览器主流程测试

### 3.2 本阶段不覆盖

- `GamePage` 三块工作台的结构改造
- 新视觉设计与布局优化
- 国家差异深化
- 海外市场深化
- 人口 / 劳动力 / 税收

## 4. 需要冻结的接口与运行时契约

### 4.1 恢复载荷

`restoreSessionContext` 返回的：

- `activeSnapshot`
- `activeGame`
- `room`
- `turnInputs`
- `gameLogs`

全部使用 2.0 `GameSnapshotPayload` 与相关 2.0 上下文字段，不允许夹带 Treasury 旧字段。

### 4.2 Socket 事件载荷

以下事件全部按 2.0 字段口径返回：

- `gamePhaseStarted`
- `gamePhaseSettled`
- `gameFinished`
- `gameSnapshotSync`
- `gamePhaseTimer`

### 4.3 Phase timer

- `phase` 只能是 `decision | market | settlement`
- 旧 `PRODUCTION / MILITARY / POLITICS` 不再出现在 timer payload、广播和测试断言里

### 4.4 Bot 与 timeout

- Bot 只允许生成：
  - `DecisionSubmission`
  - `MarketSubmission`
- timeout 自动补交也只允许补这两类提交
- `settlement` 没有玩家草稿，也没有 Bot 提交

### 4.5 前端恢复后的展示口径

恢复完成后，前端运行时和页面展示统一按以下字段工作：

- `nationalIncome`
- `budgetPools`
- `incomeAllocationRatio`
- `techPoints`
- `militaryPoints`
- `cumulativeNationalIncome`

不再展示 Treasury 主语义。

## 5. 任务群 A：恢复与持久化兼容

### 5.1 目标

把 `RecoveryRepository` 和相关恢复路径完整迁移到 2.0 快照与 2.0 状态口径。

### 5.2 文件边界

- [backend/app/modules/persistence/recovery.py](../../../backend/app/modules/persistence/recovery.py)
- `backend/app/modules/persistence/repositories.py`
- `backend/app/modules/persistence/db.py`

### 5.3 实施步骤

- 统一恢复语义到 2.0 `GameSnapshotPayload`
- 明确旧快照处理策略：
  - 已开局旧快照不做半自动迁移
  - 只允许恢复已按 2.0 保存的快照
- 刷新恢复要能恢复：
  - 当前阶段
  - 当前回合
  - 当前三池
  - 当前比例
  - 当前点数
  - 当前库存与结算摘要

### 5.4 验收标准

- 恢复结果与 2.0 页面展示口径一致
- 不再出现“恢复成功但页面仍是 Treasury 口径”的混合态

## 6. 任务群 B：实时广播与 timer

### 6.1 目标

把 Socket 事件和 timer 统一到 2.0 三阶段与 2.0 字段，不再让前端 runtime 层继续吃旧载荷。

### 6.2 文件边界

- [backend/app/modules/realtime/__init__.py](../../../backend/app/modules/realtime/__init__.py)
- [backend/app/modules/realtime/phase_timer.py](../../../backend/app/modules/realtime/phase_timer.py)
- [frontend/src/features/game/runtime/useGameRuntime.ts](../../../frontend/src/features/game/runtime/useGameRuntime.ts)
- [frontend/src/features/game/runtime/model.ts](../../../frontend/src/features/game/runtime/model.ts)
- [frontend/src/features/game/runtime/types.ts](../../../frontend/src/features/game/runtime/types.ts)
- [frontend/src/services/socket.ts](../../../frontend/src/services/socket.ts)

### 6.3 实施步骤

- 统一 `gamePhaseStarted / gamePhaseSettled / gameFinished / gameSnapshotSync` 的 payload 语义
- 修正 `gamePhaseTimer` 的 `phase` 只广播三阶段
- 前端 runtime reducer / apply 函数只按 2.0 快照字段更新状态
- 计时器与提交状态恢复继续由 runtime 层负责，但不再依赖旧阶段名

### 6.4 验收标准

- 前后端对同一事件的 payload 结构理解一致
- phase timer 测试和运行时展示不再出现旧阶段值

## 7. 任务群 C：Bot 与超时补交

### 7.1 目标

让自动补交链路和 Bot 链路与 2.0 三阶段主循环完全对齐。

### 7.2 文件边界

- `backend/app/modules/bot/planner.py`
- `backend/app/modules/bot/orchestrator.py`
- `backend/app/modules/settlement/submission_application.py`
- 相关 timeout orchestration 入口

### 7.3 实施步骤

- Bot planner/orchestrator 切到：
  - `DecisionSubmission`
  - `MarketSubmission`
- timeout orchestration 与 `SubmissionApplicationService` 对齐到 2.0 契约
- 自动补交结果进入 2.0 的 `gamePhaseSettled` 广播
- 继续保持 `settlement` 系统自动推进

### 7.4 验收标准

- Bot 不再生成旧 phase payload
- 自动补交后能正常推进到下一阶段或结束游戏

## 8. 任务群 D：测试与验收链

### 8.1 目标

把阶段 1、阶段 2 的结构改造真正变成可持续回归的 2.0 测试基线。

### 8.2 文件边界

- `backend/tests/test_phase_timer.py`
- `backend/tests/test_realtime_recovery.py`
- `backend/tests/test_timeout_orchestration.py`
- `backend/tests/test_bot_orchestrator.py`
- `backend/tests/test_settlement_realtime_orchestration.py`
- `frontend/src/test/*`
- [测试流程总览](../../测试流程总览.md)
- [本地运行与验收](../../本地运行与验收.md)

### 8.3 实施步骤

- 后端测试统一切到：
  - 三阶段 timer
  - realtime recovery
  - timeout orchestration
  - bot orchestrator
  - settlement / realtime orchestration
- 前端测试补足：
  - runtime socket apply
  - snapshot recovery
  - settlement redirect
  - ranking display
- 浏览器和本地验收至少覆盖：
  - 开局
  - 至少一整轮三阶段
  - 中途刷新恢复
  - 超时 / 自动补交
  - `15` 回合最终结算

### 8.4 验收标准

- 旧阶段断言和 Treasury 口径断言从核心测试中清退
- 本地验收流程可以直接验证阶段 3 的运行链收口结果

## 9. 本阶段禁止回退的旧模型

本阶段执行过程中，不允许再把以下概念当作正式实现方向：

- `GamePhase.PRODUCTION / MILITARY / POLITICS` 作为实时广播或 timer 输出
- Treasury 字段作为恢复载荷、排行榜或结算主口径
- `settlement` 接受玩家提交或 Bot 提交
- 前端 runtime 恢复后继续展示 Treasury 主语义

## 10. 阶段 3 完成后的移交

阶段 3 完成后，后续团队移交边界固定为：

- 2.0 主闭环已经具备：
  - 可恢复
  - 可广播
  - 可自动补交
  - 可回归
- 阶段 4 才开始接手国家差异与海外市场深化
- 阶段 5 才开始接手工业链深化与人口/税收预备
