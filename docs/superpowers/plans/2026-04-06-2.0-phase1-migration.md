# 2.0 阶段 1 实施计划

> 文档定位：阶段 1 可执行实施计划  
> 覆盖范围：共享契约与命名统一 + 后端经济内核重构  
> 不覆盖范围：前端全面工作台改造、恢复链、Bot 链、中长期系统深化

## 1. 计划目标

本计划只服务一个目的：

**把当前“新三阶段壳 + 旧 Treasury 内核”的运行时，迁移成“新三阶段 + 国家收入分账制内核”。**

本计划是 [中期计划方案书](../../中期计划方案书.md) 的第一阶段落地文档，也是 [开发任务清单](../../开发任务清单.md) 当前唯一主战场的具体执行说明。

## 2. 阶段 1 完成定义

阶段 1 完成后，系统必须满足以下条件：

1. `GamePhase` 正式只保留 `decision / market / settlement`
2. 玩家核心状态与快照正式切到国家收入制字段
3. `decision`、`market`、`settlement` 的规则执行语义与 2.0 文档一致
4. 排行与最终结算主口径改为 `cumulativeNationalIncome`
5. 旧 Treasury 主模型不再出现在核心状态对象和核心结算链里

## 3. 阶段范围与边界

### 3.1 本阶段覆盖

- 共享契约与命名统一
- 后端经济内核重构
- 平衡配置重命名与重语义
- 与以上改动直接相关的最小前端共享类型同步
- [frontend/src/types/domain.ts](../../../frontend/src/types/domain.ts) 与后端共享契约的字段对齐
- 凡是直接映射后端 payload / snapshot 的共享类型，统一归阶段 1

### 3.2 本阶段不覆盖

- 前端页面结构的全面重写
- 新工作台视觉设计与交互细节
- 刷新恢复、Socket、Bot 的全面重构
- 国家差异深化
- 海外市场深化
- 人口 / 劳动力 / 税收
- 页面层 legacy 类型清理、工作台 union 清理、草稿结构重写与 UI 文案清退
- 页面草稿、workspace union、view model、页面状态机与 UI 文案层的 legacy 清理

## 4. 目标公共接口与字段

后续共享契约统一以下目标：

### 4.1 阶段枚举

- `GamePhase = decision | market | settlement`

旧别名阶段只能被定性为历史兼容遗留，不再参与正式运行时设计。

### 4.2 玩家核心状态与快照字段

`PlayerStatePayload` 与 `GameSnapshotPayload` 统一迁移到：

- `domesticSalesRevenue`
- `overseasSalesRevenue`
- `nationalIncome`
- `cumulativeNationalIncome`
- `incomeAllocationRatio`
- `budgetPools.domesticMarket`
- `budgetPools.factory`
- `budgetPools.governmentFiscal`
- `techPoints`
- `militaryPoints`

### 4.3 提交契约

- `DecisionSubmission`
- `MarketSubmission`
- `settlement` 无玩家提交

### 4.4 排行与结算口径

- 主指标：`cumulativeNationalIncome`

### 4.5 必须清退的旧字段

- `treasuryBalance`
- `cumulativeTreasuryIncome`
- `pendingSettlementIncome`
- `domesticStrength`
- `industryStrength`
- `statecraftStrength`
- 前端 `fundPools`
- legacy 四阶段 union 类型

## 5. 任务群 A：共享契约与命名统一

### 5.1 目标

先把“名称对但语义错”的兼容层清掉，保证后续后端内核改造有稳定的公共边界。

### 5.2 文件边界

共享契约由以下文件负责：

- [backend/app/contracts/models.py](../../../backend/app/contracts/models.py)
- [backend/app/contracts/enums.py](../../../backend/app/contracts/enums.py)
- [frontend/src/types/domain.ts](../../../frontend/src/types/domain.ts)

### 5.3 实施步骤

#### A1. 清理 `GamePhase`

- 删除 legacy 四阶段别名作为正式枚举成员的存在方式
- 明确所有运行时分支只接受：
  - `decision`
  - `market`
  - `settlement`

#### A2. 重写玩家状态 payload

- 将当前 Treasury 字段替换为国家收入制字段
- 保证快照顶层和玩家状态对象语义一致
- 为收入、三池、点数、比例建立正式 schema

#### A3. 定义新的阶段提交 payload

- 定义 `DecisionSubmission`
  - 工厂决策
  - 国民消费决策
  - 政府财政决策
- 定义 `MarketSubmission`
  - 卖向
  - 卖量
  - 压仓库相关输入

#### A4. 对齐前端共享类型

- 同步 `frontend/src/types/domain.ts`
- 清退 `fundPools`
- 为阶段 2 预置 2.0 的共享类型边界
- 本任务只负责共享 schema 与字段命名对齐，不负责页面层 legacy 类型、工作台 union、草稿结构和 UI 文案清理

### 5.4 关键输出

- 新版公共 payload
- 新版阶段提交类型
- 不再混用 Treasury 字段和国家收入字段的共享模型

### 5.5 验收标准

- 共享契约层不再把 Treasury 字段当正式字段
- 新快照能独立表达：
  - 销售额
  - 国家收入
  - 三池
  - 点数
  - 比例
- 前后端类型命名一致

## 6. 任务群 B：后端经济内核重构

### 6.1 目标

把当前 Treasury 驱动的规则链，改造成 2.0 的国家收入分账制规则链。

### 6.2 文件边界

后端状态与规则执行由以下文件负责：

- [backend/app/modules/game_state/models.py](../../../backend/app/modules/game_state/models.py)
- [backend/app/modules/game_state/workspaces.py](../../../backend/app/modules/game_state/workspaces.py)
- [backend/app/modules/rules/decision.py](../../../backend/app/modules/rules/decision.py)
- [backend/app/modules/rules/market.py](../../../backend/app/modules/rules/market.py)
- [backend/app/modules/rules/settlement.py](../../../backend/app/modules/rules/settlement.py)
- [backend/app/modules/settlement/submission_application.py](../../../backend/app/modules/settlement/submission_application.py)
- `backend/config/balance/*.json`

### 6.3 实施步骤

#### B1. 重写 `PlayerState`

- 删除 Treasury 主模型字段
- 引入：
  - `domestic_sales_revenue`
  - `overseas_sales_revenue`
  - `national_income`
  - `cumulative_national_income`
  - `income_allocation_ratio`
  - `budget_pools`
  - `tech_points`
  - `military_points`

#### B2. 重写 phase workspace 语义

- `decision` workspace 直接展示三个预算池和三类主动决策
- `market` workspace 直接展示可售库存、卖向、卖量
- `settlement` workspace 直接展示：
  - 国内销售额
  - 海外销售额
  - 国家收入
  - 三池分配结果

#### B3. 重构 `decision`

- 不再“花国库 + 改 strength”
- 改为消费三个预算池中的对应预算
- 形成对本回合生产、国内市场、政府策略的状态影响

#### B4. 重构 `market`

- 不再只写 `pendingSettlementIncome`
- 改为显式产出：
  - `domesticSalesRevenue`
  - `overseasSalesRevenue`
  - `nationalIncome`

#### B5. 重构 `settlement`

- 不再回款到单一 Treasury
- 改为按 `incomeAllocationRatio` 自动分配到：
  - `budgetPools.domesticMarket`
  - `budgetPools.factory`
  - `budgetPools.governmentFiscal`
- 同时更新：
  - `cumulativeNationalIncome`

#### B6. 重构政府策略对国家结构的影响

- 政府策略可以修改下一轮 `incomeAllocationRatio`
- 比例变化按“国家结构状态变化”落库
- 不允许作为一次性奖励写入结算结果

#### B7. 重构排行与最终结算

- 排名主指标改为 `cumulativeNationalIncome`
- Treasury 相关 tie-break 从正式主逻辑中清退

#### B8. 重写 balance 配置

- 国家初始值改为三池、比例、点数、国家收入制语义
- 决策配置按：
  - 工厂决策
  - 国民消费决策
  - 政府财政决策
 进行分类

### 6.4 关键输出

- 新版 `PlayerState`
- 新版 `phaseWorkspace`
- 新版 `decision / market / settlement` 规则解析
- 新版排行与结算链
- 新版 balance 配置

### 6.5 验收标准

- 服务端核心状态不再依赖 Treasury 模型
- `market` 结果能直接表达国内销售额、海外销售额、国家收入
- `settlement` 能按比例把国家收入重新分配到三池
- 政府策略能修改下一轮收入结构
- 最终排名按 `cumulativeNationalIncome`

## 7. 实施顺序

阶段 1 的执行顺序固定如下：

1. 先完成任务群 A，冻结共享契约与命名
2. 再进入任务群 B，重构后端状态和规则执行
3. 最后统一修正排行、最终结算与配置
4. 阶段 1 完成后，才允许阶段 2 正式开工

## 8. 验证清单

### 8.1 契约验证

- `GamePhase` 只剩 `decision / market / settlement`
- `PlayerStatePayload` 和 `GameSnapshotPayload` 不再包含 Treasury 主字段
- `DecisionSubmission` 与 `MarketSubmission` 已落地

### 8.2 后端流程验证

- `decision` 能按三池消耗预算
- `market` 能产出国内销售额、海外销售额、国家收入
- `settlement` 能按比例重新分配三池
- 排名按 `cumulativeNationalIncome`

### 8.3 配置验证

- 初始国家配置与 2.0 规则一致
- 决策目录配置按三类主动决策分类

## 9. 本阶段禁止回退的旧模型

本阶段执行过程中，不允许再把以下概念当作正式实现方向：

- Treasury 作为唯一底层真钱
- `pendingSettlementIncome` 作为市场阶段唯一结算产物
- `domesticStrength / industryStrength / statecraftStrength` 作为 2.0 主状态替代品
- legacy 四阶段 union 与 legacy `fundPools`

## 10. 阶段 1 完成后的移交

阶段 1 是否冻结，统一按以下 gate 判定：

1. 新 `GameSnapshotPayload` 已稳定
2. 新 `DecisionSubmission / MarketSubmission` 已稳定
3. 第 8 节验证清单全部通过
4. 由技术负责人确认后，才允许切入阶段 2

阶段 1 完成后，后续团队移交边界固定为：

- [开发任务清单](../../开发任务清单.md) 继续作为项目级推进依据
- [前端产品化开发任务清单](../../前端产品化开发任务清单.md) 正式进入 `P1` 到 `P5`
- 恢复、实时、Bot 与测试链在阶段 3 统一接入，不提前并行打散
