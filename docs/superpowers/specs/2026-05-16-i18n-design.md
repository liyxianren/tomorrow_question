# i18n Internationalization Design

## Overview

Add comprehensive i18n support to the entire project (frontend UI + backend presentation config + API messages), with English as the default language and Chinese as a secondary language.

## Scope

- **Frontend**: All user-facing text in React components (~190 strings across 80+ files)
- **Backend presentation config**: `phases.json` and `countries.json` language variants
- **Backend API messages**: Error messages and hints

## Tech Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Frontend library | react-i18next | Largest ecosystem, TS support, auto-extraction tools |
| Language detection | browser → localStorage → fallback en | Standard pattern, user preference persists |
| Namespace structure | common / game / lobby / room / pages | Separates concerns, lazy-loadable |
| Backend language | `Accept-Language` header | Follows HTTP standard |

## Project Structure

### Frontend additions

```
frontend/src/i18n/
├── index.ts                  # i18next init + detection
├── resources/
│   ├── en/
│   │   ├── common.json       # Shared UI (buttons, status, errors)
│   │   ├── game.json         # Game concepts (phase names, countries, tech, ideologies)
│   │   ├── lobby.json        # Lobby page
│   │   ├── room.json         # Room page
│   │   └── pages.json        # Page-specific text (home, settlement, etc.)
│   └── zh/
│       └── ...               # Chinese translations (same structure)
```

### Backend changes

```
backend/config/presentation/
├── phases.en.json            # English phase descriptions
├── phases.zh.json            # Chinese phase descriptions
├── countries.en.json         # English country contexts
├── countries.zh.json         # Chinese country contexts
backend/app/
├── i18n.py                   # Simple dict mapping for API messages
```

## Key Migration Strategy

1. Build complete Chinese JSON dictionaries first (extract from existing code)
2. Create English translations for each key
3. Replace hardcoded text in components with `t('ns:key')` calls
4. `labels.ts` / `panelGlossary.ts` functions retain signatures but use i18n internally — callers need minimal changes
5. Backend presentation config split into language variants, selected via `Accept-Language`

## Language Detection Priority

```
browser navigator.language → localStorage "app_locale" → fallback "en"
```

## Dictionary Key Convention

Nested object path keys: `game.phase.decision`, `lobby.roomCard.hostLabel`. No flat keys.

## Component Usage Pattern

```tsx
const { t } = useTranslation('game');
<span>{t('phase.decision')}</span>
```

## Language Switcher

Toggle button on SettingsPage. Calls `i18n.changeLanguage()`, persists to localStorage, updates Accept-Language for subsequent API calls.
