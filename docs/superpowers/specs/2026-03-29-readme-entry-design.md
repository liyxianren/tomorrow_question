# README 入口文档设计说明

## 1. 背景

当前项目处于需求方案已确认、准备直接进入开发的阶段。仓库内现有资料主要为需求与交付确认文档，缺少一份面向开发团队的统一入口文档，用于快速说明项目目标、开发边界、系统结构和后续启动方式。

本次设计目标是产出项目根目录 `README.md`，作为开发团队进入项目时的第一份说明文档，而不是替代完整规则文档或游戏设计文档。

## 2. 文档定位

`README.md` 的目标读者为开发团队。

文档定位为“开发启动入口”，主要解决以下问题：

- 这个项目是什么
- 本轮开发要交付什么
- 系统应该如何拆分理解
- 开发应优先按什么顺序推进
- 哪些文档是正式开发依据

文档不承担以下职责：

- 不作为完整规则手册
- 不作为美术或交互风格规范
- 不维护待确认事项列表
- 不沉淀详细数值设计过程

## 3. 设计原则

- 以开发视角组织内容，而不是以产品宣传视角组织内容
- 以“本次开发范围”和“模块边界”优先，避免 README 变成规则全文复制
- 直接承接现有确认文档，不引入新的待确认项
- 明确“最终交付确认文档”为最高优先级依据
- 保持足够详细，能支持团队启动开发，但不展开到实现细节级别

## 4. README 结构设计

### 4.1 项目概述

说明项目背景、题材、产品形式和当前阶段，明确这是一个支持 PC 与移动端浏览器访问的多人网页回合策略游戏，并指出当前已进入可直接开发阶段。

### 4.2 本次开发目标

明确本轮开发要交付的核心目标：

- 可访问的网站游戏
- 固定 5 人房间制对局
- 15 回合完整流程
- 五大核心系统闭环
- 可玩的 MVP 版本

### 4.3 本次开发范围

汇总本次需要覆盖的功能与系统：

- 在线房间与开局流程
- 国家差异化设定
- 生产、科技、市场、军事、政治五大系统
- 地图连接、海洋节点、封锁与控制逻辑
- 回合推进、结算、日志、排行榜
- 核心数值配置结构

### 4.4 核心玩法与系统主干

从开发视角总结回合制流程和系统耦合关系，强调以下主链路：

生产 -> 市场投放 -> 收入分配 -> 三资金池变化 -> 建设 / 科研 / 军政投入 -> 政策与状态反向影响下回合。

这一部分用于帮助开发团队快速把握规则引擎的主数据流和控制流。

### 4.5 页面与模块职责

按页面和模块两个维度描述系统边界：

- 房间页
- 选国页
- 游戏主界面
- 五大系统操作区
- 日志与排行榜
- 结算页

同时补充推荐的逻辑模块划分：

- 房间与对局编排
- 回合状态机与结算引擎
- 规则域模块
- 地图与连接模型
- 数值配置模块
- 前端表现层

### 4.6 开发实现原则

写清当前开发阶段的共识原则：

- 规则引擎与 UI 解耦
- 同步提交、统一结算
- 数值配置化
- 日志可追踪
- 优先实现完整可玩闭环，再增强表现层

### 4.7 文档入口

列出现有三份文档及用途，强调：

- `word/最终交付确认文档.md` 为最高优先级依据
- 其他两份文档作为规则背景与需求澄清补充

### 4.8 建议开发顺序

给出开发启动建议顺序：

1. 工程骨架与基础约定
2. 对局状态模型与回合状态机
3. 五大系统规则实现
4. 主界面与操作流
5. 日志、排行与结算
6. 数值调试与联调

## 5. 风险与边界控制

README 中应显式提醒以下内容，但不展开成完整风险分析：

- 五大系统高度耦合，规则引擎设计必须先行
- 当前阶段以“开发确认版文档”作为最终范围依据
- 数值需要配置化，否则后续平衡和迭代成本会很高
- 由于时间有限，优先保证可玩闭环，不追求复杂表现层

## 6. 实施说明

将按本设计在项目根目录新增 `README.md`。文档语言默认使用中文，保留路径、模块名和系统标识的英文命名空间表达方式，便于后续工程文件直接对应。
