# 测试流程总览

## 1. 测试目标

测试的目标是验证国家收入制的正式规则能否稳定工作。

测试必须覆盖：

- 三阶段状态流
- 三个预算池扣减与分配
- 科技点 / 军事点购买与消耗
- 国家收入结算
- 比例变化
- 刷新恢复
- `15` 回合最终排名

## 2. 测试层级

## 2.1 阶段收口顺序

测试链的推进顺序固定为：

1. 阶段 1 先冻结共享契约和后端经济内核
2. 阶段 2 再完成前端运行时与工作台
3. 阶段 3 最后统一接管：
   - 刷新恢复
   - Socket 广播
   - phase timer
   - Bot 自动补交
   - 超时补交
   - 前端运行时测试
   - 浏览器主流程测试

阶段 3 是 2.0 测试与验收链的正式收口阶段，不与阶段 2 并行实现。

### 单元测试

重点验证：

- 国家收入计算
- 三池分账
- 工厂动作成本与产能变化
- 国民消费动作对国内市场能力的影响
- 政府动作对比例与点数的影响

### 接口测试

重点验证：

- `decision` 提交
- `market` 提交
- `settlement` 自动推进
- 快照返回字段完整
- 恢复接口返回状态正确
- Socket 事件 payload 与阶段 timer payload 正确

### 前端测试

重点验证：

- 三阶段面板切换
- 三池与点数展示
- 决策表单与卖货表单
- 财政结算摘要展示
- 运行时恢复与 Socket 事件应用

### 浏览器测试

重点验证：

- 创建房间
- 五人开局
- 进入三阶段对局
- 刷新恢复
- 超时 / 自动补交
- 最终结算展示

## 3. 建议执行顺序

### 正式签收门槛

```powershell
./scripts/run-test-suite.ps1
```

正式签收默认要求：

- `./scripts/run-tests.ps1`
- `./scripts/run-browser-tests.ps1 -Mode full`
- `./scripts/check-local.ps1 -Mode full`
- `./scripts/run-test-suite.ps1`

其中：

- `run-browser-tests.ps1 -Mode smoke` 只用于快速冒烟
- `run-browser-tests.ps1 -Mode full` 才是浏览器交付门槛
- `check-local.ps1 -Mode smoke` 用于快速联调
- `check-local.ps1 -Mode full` 才是 HTTP / 页面联调验收门槛

### 快速检查

```powershell
./scripts/run-tests.ps1
```

### 本地完整套件

```powershell
./scripts/run-test-suite.ps1
```

### 启动本地环境后做 HTTP/页面检查

```powershell
./scripts/start-local.ps1
./scripts/check-local.ps1 -Mode smoke
./scripts/check-local.ps1 -Mode full
```

### 浏览器测试

```powershell
./scripts/run-browser-tests.ps1 -Mode smoke
./scripts/run-browser-tests.ps1 -Mode full
```

## 4. 验收重点

每轮规则改动后，至少要确认：

- 第一阶段一定包含：
  - 工厂决策
  - 国民消费决策
  - 政府财政决策
- 第二阶段一定可表达：
  - 卖向
  - 卖量
  - 压仓库风险
- 第三阶段一定会输出：
  - 国内销售额
  - 海外销售额
  - 国家收入
  - 三池重新分配
- 最终结算一定按：
  - `15` 回合累计国家收入

## 5. 回归清单

- 创建房间并凑满 `5` 人
- 完整进入第 `1` 回合
- 跑过至少一整轮三阶段
- 验证政府策略能改变收入结构
- 刷新任意玩家页面并恢复
- 验证刷新后 `round / phase / submissionStatus / latestSettlement` 与恢复接口一致
- 验证超时补交与 Bot 自动补交能继续推进回合
- 跑满 `15` 回合并得到最终排名
- 验证最终排名顺序与 `/final-result` 返回的 `cumulativeNationalIncome` 一致
