# 接手开发说明

> 文档定位：内部工程交接入口  
> 使用对象：新接手的工程师 / 技术负责人  
> 说明：本文件只负责导航与优先级说明，不替代正式规则文档与实施计划

## 1. 文档定位

本文件是项目的**开发交接入口**，不是客户文档，也不是规则总纲。

它只回答五个问题：

1. 哪些文档是权威文档
2. 当前项目真实停留在哪个阶段
3. 接手后第一步必须从哪里开始
4. 阶段 1、阶段 2、阶段 3 分别对应哪一份实施计划
5. 哪些旧语义已经不能再继续使用

如果本文件与正式规则或技术文档冲突，以正式文档为准，不以本文件为最终真相。

## 3. 当前项目真实状态

当前仓库的正式规则、技术命名和任务体系，已经统一到 2.0 的**国家收入制**：

`国家决策 -> 市场出售 -> 财政结算`

但代码真实状态仍然是：

- 阶段外壳已经切到三阶段
- 共享契约、后端结算、前端工作台、恢复链和测试链仍处于新旧混合态
- Treasury 旧语义尚未从运行时完全清退

更直接地说：

**文档真相已经是 2.0，但代码运行时还没有完全变成 2.0。**

所以接手后不能按“功能迭代项目”来理解，而必须按“运行时语义迁移项目”来理解。

接手当天建议先完成以下 baseline checklist：

- 若仓库处于 Git 管理下，先确认当前分支与工作区状态
- 检查后端核心契约文件是否仍以 Treasury 旧字段为主
- 检查 [frontend/src/types/domain.ts](../frontend/src/types/domain.ts) 是否仍未完成 2.0 对齐
- 对照阶段 1 实施计划，确认目标字段和目标 payload 是否尚未落地

首日建议优先检查以下代码入口：

- [backend/app/contracts/models.py](../backend/app/contracts/models.py)
- [backend/app/modules/game_state/models.py](../backend/app/modules/game_state/models.py)
- [frontend/src/types/domain.ts](../frontend/src/types/domain.ts)

## 4. 当前必须执行的阶段

当前正式开发起点固定为：**阶段 1**

也就是说：

- 阶段 1 是当前唯一允许正式开工的主战场
- 阶段 2、阶段 3 虽然已经有实施计划，但它们是后续严格串行阶段，不能抢跑
- 阶段 4、阶段 5 目前只保留总控级规划，不进入施工单

## 6. 禁止继续使用的旧语义

接手后，以下旧语义不能再继续作为正式实现方向：

- `treasuryBalance`
- `cumulativeTreasuryIncome`
- `pendingSettlementIncome`
- `domesticStrength`
- `industryStrength`
- `statecraftStrength`
- `production / military / politics`
- `fundPools`

如果代码里仍然存在这些字段或阶段名，应把它们理解为：

- 历史遗留
- 兼容态残留
- 待迁移对象

而不是未来继续扩展的基础。

## 7. 仓库模块入口

接手时只需要先把几个入口目录职责看清：

- `backend/`
  - Flask 后端、规则执行、状态快照、持久化、实时链、Bot
- `frontend/`
  - React 前端、对局页面、运行时、工作台、结算页
- `docs/`
  - 正式规则、技术设计、任务树、实施计划
- `scripts/`
  - 本地运行、检查、测试、联调辅助脚本

这里的原则是：

- 先用 `docs/` 建立规则与阶段认知
- 再进入 `backend/` 和 `frontend/` 做实现判断
- 最后用 `scripts/` 做验证与验收

## 8. 交接结论

如果你是新的接手工程师，当前最重要的判断不是“接下来要不要做新功能”，而是：

**先完成 2.0 运行时迁移，再谈后续扩展。**

因此，接手后的第一动作应该是：

- 先读规则与技术文档
- 确认当前真实起点是阶段 1
- 以阶段 1 实施计划作为当前唯一开发依据

在阶段 1 完成之前，不要把阶段 2、阶段 3 当作并行开发包，更不要提前进入阶段 4、阶段 5。
