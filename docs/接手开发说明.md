## 2. 当前系统真实状态

当前系统已经形成完整主链路：

```text
首页 / 大厅 / 房间 -> 选国 / Ready / AI补位 -> 自动开局
-> 国家决策 -> 市场出售 -> 财政结算 -> 下一回合
-> 最终结算 / 排名
```

当前应按“可运行系统 + 核心经济模型待重构”理解，而不是按旧文档中的“2.0 迁移施工中”理解，也不能把当前 Demo 的随机 / 简化价格模型当作验收目标。

## 2.1 会后优先事项

2026-04-27 会议后，优先级调整为：

1. 先以 [用户原始需求：核心机制](./用户原始需求-核心机制.md) 作为原始复杂机制入口。
2. 以供给、需求、价格形成机制重构工业区与市场。
3. 暂时保留议会厅、军事要塞、研究院与现有 UI 框架。
4. 不再把当前 Demo 的经济模型作为核心验收依据。

## 3. 当前核心模块

- `frontend/`
  - 首页、大厅、房间、游戏主页面、决策面板、贸易港、结算页
- `backend/`
  - 房间、会话、游戏快照、阶段提交、结算、Bot、恢复、Socket
- `docs/`
  - 当前系统说明、用户需求源、玩家规则、工程文档、历史过程文档
- `alignment-site/`
  - 会议用“真实游戏截图 × 需求文档”静态对齐页

## 4. 文档优先级

当前冲突处理顺序：

1. [第一版本需求确认文档](./第一版本需求确认文档.md)
2. [最终交付确认文档](./最终交付确认文档.md)

说明：

- 当前系统说明用于判断“现在系统实际长什么样”。
- 历史过程文档只能解释开发过程，不再作为当前实现方向的唯一依据。

## 5. 接手第一步

接手前先做三件事：

1. 检查 Git 状态，避免把历史临时文件混入提交。
2. 跑本地前后端，确认首页、大厅、房间、游戏页可访问。
3. 按 [本地运行与验收](./本地运行与验收.md) 与 [测试流程总览](./测试流程总览.md) 做最小验收。

## 6. 不要再使用的旧判断

以下旧判断已经不适合作为当前项目入口口径：

- “代码运行时还没有完全变成 2.0”
- “当前唯一允许正式开工的是阶段 1 迁移”
- “阶段 2 / 阶段 3 不能抢跑”
- “Treasury 旧语义仍是主要运行态”

如果这些说法仍出现在历史文档中，应理解为当时开发阶段的记录，不是当前事实。

客户或用户沟通时，建议先展示：

1. 真实游戏

3. `alignment-site/index.html` 里的真实截图对齐页
