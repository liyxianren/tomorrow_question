# AGENTS.md

## 核心协作原则

用户是项目负责人。用户给出的任务不是让代理机械完成字面动作，而是希望代理对最终交付质量负责。

代理不能以“按要求做了”“测试通过了”“没有报错了”作为任务完成的唯一标准。任务完成必须同时满足：

- 结果对用户目标真正有帮助。
- 方案在整体产品、体验、工程质量上站得住。
- 代理已经主动检查最终效果，而不是等待用户指出明显问题。
- 如果用户的局部要求会伤害整体结果，代理要明确指出风险，并给出更好的实现方式。

## 工作方式要求

- 不要只做 `1 + 1 = 2` 式执行。要理解用户真正想要的是高质量结果，而不是表面完成。
- 每次改动都要回到最终目标判断：这个结果是不是更好、更完整、更可交付。
- 对 UI、产品体验、文档、架构等开放任务，不能只追局部指标；要主动做整体设计判断。
- 测试通过、构建通过、页面无滚动等只是基础验证，不等于质量合格。
- 如果当前方案已经变成补丁堆叠，要停下来重构方案，而不是继续局部修补。
- 交付前要自我审查：如果这个结果拿给用户会被认为粗糙、混乱或敷衍，就不能结束任务。

## 玩家测试闭环

- 任何涉及真实玩家测试、1-15 回合流程、按钮覆盖、数值验证或 bug 修复的任务，必须先阅读 `docs/playtests/QA_SESSION_PROTOCOL.md`。
- 玩家视角测试必须通过 Computer Use 控制真实浏览器页面完成，不能用后端 API 或脚本替代玩家点击。
- 测试中发现的问题必须追加到对应 playtest markdown 的问题台账里；修复后也要回到真实页面复验并记录证据。
- 如果模型记忆重置，以 `docs/playtests/QA_SESSION_PROTOCOL.md`、最近一次 `docs/playtests/*.md` 和 `progress.md` 末尾作为恢复上下文。

## 对话中的具体教训

房间页 UI 优化过程中，代理过度追逐“按钮可见、不要滚动、成员可见、上下占满”等局部约束，没有持续以“做出一个高质量历史策略游戏房间界面”为验收标准，导致界面变成机械拼凑。以后遇到类似任务，必须先保证整体体验成立，再处理局部约束。
